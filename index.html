ICA Agent — Data Consumption & Dashboard Display Capabilities

Everything the ICA "AI Vulnerability Remediation" agent returns can be captured and displayed here. The table below documents exactly what information flows from the agent into this dashboard.

Data Category What the ICA Agent Returns How It Is Displayed Limit / Constraint
🔍 Vulnerability Description Concise plain-language explanation of the vulnerability (CWE, type, impact) Formatted text section in pinned card Full text — no truncation
🧬 Root Cause Analysis Technical explanation of why the code is vulnerable (e.g. hard-coded string in source) Formatted text section in pinned card Full text — no truncation
🔧 Remediation Steps Numbered action steps to fix the vulnerability Bulleted list in pinned card Full list — no truncation
💻 Sample Code Fix Before/after code snippet in the relevant language (Java, Python, etc.) Syntax-highlighted code block (dark theme) Full code — no truncation
🛡️ Secure Coding Best Practices 5–7 preventative recommendations relevant to the CWE category Formatted list in pinned card Full list — no truncation
📎 Uploaded File Context Excel/CSV rows injected as context. Up to 12 000 characters of file content per request Referenced in prompt; response reflects the data 12 000 chars (~150–200 rows of data)
💬 Conversation History Multi-turn chat — agent remembers context within a session (session_id) Chat panel retains last 40 messages (20 turns) Last 40 messages kept in memory
📌 Pinned Cards Any agent response can be pinned to this tab as a structured card Up to N cards pinned simultaneously; each removable No hard limit — browser memory only
🏷️ ICA Session ID Returned by ICA API to maintain conversation continuity Stored in JS memory; sent with every follow-up request Per-browser-session (lost on refresh)

* The 12 000-character file injection limit can be increased by editing uploadedFileText.slice(0, 12000) in the dashboard JS. Claude 4.5 Sonnet supports ~180 000 tokens of context — well beyond any single vulnerability report.

This HTML file is fully self-contained. It uses SheetJS to parse any .xlsx or .csv file directly in the browser — no server, no upload to external systems, no manual copy-paste. Every time the Polaris export changes, simply upload the new file and the dashboard regenerates instantly.

IBM Bob Integration (Playground Workflow)
  • Step 1: Copy your Polaris .xlsx export into C:\Users\002K6E744\.bob\playground\
  • Step 2: Tell Bob: "Process my vulnerability file vulnerabilities.xlsx"
  • Step 3: Bob uses read_xlsx to read all rows and columns automatically
  • Step 4: Bob calls generate_batch_report via the Polaris Remediation MCP Server
  • Step 5: Bob renders this dashboard with full remediation guidance, code fixes, and action plans
IBM ICA (Watsonx Assistant) Integration
  • Register Bob as an ICA Action: Expose the "Process Vulnerability File" workflow as an ICA skill
  • Developer Workflow: Developer opens ICA chat → uploads or references the Polaris export → ICA triggers Bob → Bob reads xlsx, runs MCP, returns this report in chat
  • No Security Expertise Required: Developers get contextual, file-specific remediation guidance without needing to understand SAST findings manually
  • Scalable: Works for 10 findings or 10,000 — Bob batches MCP calls automatically
  • Audit Trail: Every report is timestamped and linkable to the source Polaris finding IDs
Supported Input Formats
  • .xlsx / .xls — Polaris Excel export (any sheet, any row count)
  • .csv — Polaris CSV export with the same column headers
  • Variable schema: The dashboard auto-detects column positions by name — column order does not matter
  • New CWEs: The remediation engine auto-generates guidance for any CWE not in its built-in library
